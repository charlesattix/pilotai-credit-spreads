2. Problems with Current Approach

🔴 P1 — No Experiment Registry
There's no central place that knows which experiments exist, what state they're in, or how to compare them. Configs are manually named files. Starting/stopping experiments requires remembering CLI commands. No way to list "what's running right now?"
🔴 P2 — Alpaca Account Collision
All experiments share the same Alpaca paper account. If EXP-400 opens a bull put spread on SPY and EXP-401 opens a bear call spread, neither knows about the other's positions. The PositionMonitor in EXP-400 might try to manage EXP-401's positions. Worse: PositionReconciler runs on startup and syncs all Alpaca positions into the local DB — cross-contaminating everything.
🔴 P3 — No Process Orchestration
Each experiment is a standalone python main.py scheduler process. There's no supervisor to: (a) start all experiments together, (b) restart crashed ones, (c) stop all gracefully, (d) show status of all experiments, (e) coordinate shared resources.
🟡 P4 — No Centralized Comparison Dashboard
Each experiment writes to its own DB. To compare performance across experiments, you'd need to manually query each DB and join the results. There's no unified view of "EXP-400 is up 12% and EXP-401 is up 8% this week."
🟡 P5 — Configuration Drift
Configs are manually created YAML files with 200+ lines. Easy to forget a field, use stale defaults, or create configs that pass validate_config() but don't match the backtested parameters. No config inheritance or templates.
🟡 P6 — Heartbeat & Healthcheck Collision
The heartbeat file (data/heartbeat.json) and the healthcheck HTTP server (port 8080) are singletons. Multiple experiments overwrite each other's heartbeats. Multiple healthcheck servers can't bind the same port.

4. Component Design

4.1 Experiment Orchestrator (pilotctl)

A CLI tool and Python module that manages the experiment lifecycle. Think of it as docker-compose for experiments.

# Create a new experiment from a template
pilotctl create --name "EXP-402" \
    --base champion.json \
    --override "risk.max_risk_per_trade=10" \
    --override "strategy.straddle_strangle.enabled=false"

# List all experiments
pilotctl list

# View logs

# Nuke an experiment (DB, logs, config — with confirmation)

6. Isolation Model

What MUST Be Isolated (Per-Experiment)

ResourceIsolation StrategyImplementation
Database Separate SQLite file per experiment data/pilotai_exp{id}.db — already implemented (Bug #1 fix)
Log Files Separate log file per experiment logs/exp_{id}.log — auto-set by Config Factory
Heartbeat Separate heartbeat file data/heartbeat_exp{id}.json — needs scheduler change
Alert Files Namespaced output files output/alerts_exp{id}.{json,csv,txt}
Trade Tagging All orders tagged with experiment ID client_order_id prefix: exp400-cs-abc123
Position Filtering Each experiment only manages its own trades Filter by client_order_id prefix in PositionMonitor

What CAN Be Shared (Centralized)

ResourceWhy ShareableNotes
Polygon Data Cache Read-only market data, idempotent Saves API calls; safe for concurrent reads
Alpaca Credentials One paper account, order-tagged Use client_order_id prefix to partition
Economic Calendar Static reference data FOMC/CPI/NFP dates don't change
VIX/Regime Data Same market regime for all Could cache centrally, but per-process is fine too

🔑 The Alpaca Problem — Solved with Order Tagging

Solution: Prefix all client_order_id values with the experiment ID.

Currently: cs-a1b2c3d4e5f6g7h8
Proposed: exp400-cs-a1b2c3d4e5f6g7h8

Then the PositionMonitor and PositionReconciler filter by prefix: 
# position_monitor.py — only manage our experiment's trades
orders = alpaca.list_orders(status='all')
my_orders = [o for o in orders 
             if o.client_order_id.startswith(f"{self.exp_id}-")]
This lets multiple experiments safely share one Alpaca paper account without cross-contamination.

7. Experiment Lifecycle

CREATE START STOP ARCHIVE │ │ │ │ ▼ ▼ ▼ ▼ ┌────────┐ ┌─────────┐ ┌─────────┐ ┌──────────┐ │CREATED │──start──▶ │ RUNNING │──stop───▶ │ STOPPED │──arch──▶ │ ARCHIVED │ │ │ │ │ │ │ │ │ │• Config│ │• Process│ │• DB kept│ │• DB kept │ │ gen'd │ │ alive │ │• Config │ │• Frozen │ │• DB │ │• Scans │ │ kept │ │• Compare │ │ init'd│ │ active │ │• Can │ │ only │ │ │ │• Trades │ │ restart│ │ │ └────────┘ └────┬────┘ └─────────┘ └──────────┘ │ crash│restart │ ┌────▼────┐ │ CRASHED │──auto──▶ RUNNING (with backoff) │ │ │• Logged │ │• Alert │ │ sent │ └─────────┘

Lifecycle Operations

  1. CREATE — Config Factory generates YAML from base + overrides. Registry entry created. DB initialized (empty). Validated.
  2. START — Process Manager spawns python main.py scheduler. PID recorded. Heartbeat monitoring begins.
  3. RUNNING — Scheduler scans on market schedule. Trades executed. Positions monitored. Heartbeat written every scan.
  4. STOP — SIGTERM sent. Process exits gracefully. Status updated. DB preserved for analysis.
  5. RESTART — Stop then start. Dedup persistence ensures no duplicate trades on restart.
  6. ARCHIVE — Experiment frozen. No more trades. DB and logs preserved. Available for comparison.
  7. DESTROY — Everything deleted (DB, logs, config). Requires --confirm flag.

Key Design Decisions

DecisionChoiceRationale
Process model Separate OS processes (not threads) Full isolation: if one experiment crashes, others survive. SQLite WAL handles concurrent reads. No GIL contention.
Config management Template + overrides (not manual YAML) Eliminates config drift. One source of truth for defaults. Easy to create variations.
Alpaca sharing One account, order-tagged Alpaca paper accounts are limited. Tagging is deterministic and filterable.
Registry storage SQLite (not YAML/JSON) Queryable, atomic updates, supports concurrent access from CLI + orchestrator.
Supervision Built-in Python (not systemd/supervisor) Portable. Runs on Mac, Linux, Docker. No external dependencies.
Comparison Read from experiment DBs directly No data duplication. Each experiment's DB is the source of truth. Comparison is read-only.

Total estimated effort: ~6 days of focused development.

After this, adding a new paper trading experiment becomes a one-liner:

Fully isolated. Auto-supervised. Auto-compared. Production-grade.
