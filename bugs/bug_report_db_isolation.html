🐛 Bug Report: DB Isolation & Dedup Failures

PilotAI Credit Spreads — Filed by Charles · March 13, 2026 · Branch: maximus/champion-config

Three related bugs preventing multi-experiment paper trading from working correctly. All stem from the same root cause: the YAML db_path config field is never read by the application.

Bug #1: DB Isolation Broken CRITICAL

Summary

Every experiment writes to the shared data/pilotai.db regardless of what db_path is set in the YAML config. This means experiments cross-contaminate each other's trade records, dedup state, and position tracking.

Evidence

Config file paper_exp401.yaml specifies:

db_path: "data/pilotai_exp401.db"

But the scheduler logs show:

Database initialized at /Users/charlesbot/projects/pilotai-credit-spreads/data/pilotai.db

Root Cause

The application reads DB path from two sources only:

  1. --db-path CLI argument → sets os.environ['PILOTAI_DB_PATH']
  2. PILOTAI_DB_PATH environment variable (checked at module import time)

The YAML db_path field is never read. Here's the chain:

# shared/database.py:19 — hardcoded fallback at import time
DB_PATH = Path(os.environ.get('PILOTAI_DB_PATH', str(Path(DATA_DIR) / "pilotai.db")))

# main.py:824-826 — only CLI arg sets the env var
if args.db_path:
    os.environ['PILOTAI_DB_PATH'] = args.db_path

# The YAML config['db_path'] is NEVER passed to any component

All downstream components (ExecutionEngine, PositionMonitor, AlertRouter, PositionReconciler) receive db_path=os.environ.get('PILOTAI_DB_PATH'), which defaults to pilotai.db.

Impact: All experiments share one database. Trade records from exp400 and exp401 co-mingle. PositionMonitor in one experiment can see and act on trades from another. P&L tracking is unreliable. This defeats the entire purpose of isolated experiments.

Proposed Fix

In main.py:create_system(), after load_config(), read the YAML db_path and set the env var: 
config = load_config(config_file, env_file=env_file)
if not os.environ.get('PILOTAI_DB_PATH') and config.get('db_path'):
    os.environ['PILOTAI_DB_PATH'] = config['db_path']
This respects CLI arg priority (already set) but falls back to YAML config before the hardcoded default.
Current workaround: Pass --db-path data/pilotai_exp401.db on the command line, or add PILOTAI_DB_PATH=data/pilotai_exp401.db to the .env.exp* file.

Bug #2: alert_dedup Table Missing direction Column HIGH

Summary

The AlertRouter tries to persist dedup entries with a direction column, but the alert_dedup table schema doesn't include it. This causes a non-fatal warning on every scan and prevents cross-restart dedup from working.

Evidence

AlertRouter: could not persist dedup entry (non-fatal): table alert_dedup has no column named direction
AlertRouter: could not load dedup entries from DB (non-fatal): no such column: direction

Root Cause

The dedup persistence code (added in BUG #17 fix) calls upsert_dedup_entry(ticker, direction, ...) which writes to the alert_dedup table. But the DB migration that creates/alters this table never added the direction column. The table was likely created before the direction-aware dedup was implemented.

Consequence

Without persistent dedup, every scheduler restart resets the dedup ledger. Combined with Bug #3, this means the same trade gets resubmitted every scan after a restart.

Fix: Add a migration in shared/database.py's init_db(): 
ALTER TABLE alert_dedup ADD COLUMN direction TEXT DEFAULT '';
Or update the CREATE TABLE statement if the table is created fresh.

Bug #3: client_order_id Collision on Resubmit HIGH

Summary

The ExecutionEngine generates a deterministic client_order_id from trade fields. If the same opportunity appears on the next scan (because dedup failed — see Bug #2), it tries to submit with the same client_order_id. Alpaca rejects it with:

APIError: {"code":40010001,"message":"client_order_id must be unique"}

Root Cause

The client_order_id is built deterministically:

# execution/execution_engine.py:80-81
raw_id = f"{ticker}-{spread_type}-{expiration}-{short_strike}-{long_strike}"
client_id = "cs-" + hashlib.sha256(raw_id.encode()).hexdigest()[:16]

This is intentionally deterministic for idempotency. The problem isn't here — it's that the dedup layer (Bug #2) fails to prevent the resubmission in the first place.

Consequence

Every scan that finds the same opportunity logs an error and fails to submit. The existing position is safe (Alpaca rejects duplicates), but the error log is noisy and masks real issues. For iron condors, if the put wing succeeds but the call wing resubmit fails, the IC is incomplete.

Fix: Fixing Bug #2 (dedup persistence) eliminates this. As defense-in-depth, the ExecutionEngine should also check if a trade with this client_order_id already exists in the DB before submitting: 
# Before submitting to Alpaca:
existing = get_trade_by_client_id(client_id, path=self.db_path)
if existing and existing['status'] not in ('rejected', 'cancelled'):
    logger.info("Trade %s already exists (status=%s), skipping", client_id, existing['status'])
    return {"status": "duplicate", "client_order_id": client_id}
